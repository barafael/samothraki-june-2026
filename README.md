# Samothraki Holiday

A [Dioxus](https://dioxuslabs.com/) app that plots holiday photos and videos on
a map. One codebase, two build profiles:

| Profile    | Feature            | Fullstack? | Data source          | Use                                   |
| ---------- | ------------------ | ---------- | -------------------- | ------------------------------------- |
| **Editor** | `editor`           | yes        | local disk + server fns | edit metadata / annotate GPS locally |
| **Viewer** | `web` (default)    | no         | `manifest.json`      | the deployed, fully static site       |

The **viewer** WASM contains no `#[server]` machinery — all photo data is
fetched at runtime from `manifest.json`, and images load from a configurable
asset base URL (Cloudflare R2 in production).

## Where the media lives

Photos and videos are **not in git**. Local disk is the source of truth; R2 is
the published copy.

- `Photos-3-001/` — original photos and (HEVC) videos. Gitignored.
- `media-web/` — browser-playable H.264 transcodes of the videos, produced by
  `scripts/transcode_videos.sh`. Gitignored.
- `thumbs/`, `manifest.json` — generated by `photo-extract gen-manifest`.
  Gitignored.

Curated metadata **is** in git: `assets/photo_data.json` (photo entries +
GPS, including manual annotations) and `assets/photos_no_gps.json` (the
editor's to-annotate list).

## Edit locally (editor profile)

Runs the fullstack app on your machine. Server functions read/write
`assets/photo_data.json` and serve photos straight from `Photos-3-001/`.
Nothing here is ever deployed.

```sh
dx serve --features editor --fullstack
```

- The **Map** tab shows every geotagged photo; click a marker to view it.
- The **Annotate** tab assigns GPS coordinates to photos that lack them
  (pick a point on the map or type lat/lng); saves are written back to disk.

To (re)extract EXIF/ffprobe metadata from scratch into `photo_data.json`:

```sh
cargo run -p photo-extract --bin extract
```

## Publish (viewer profile)

`scripts/publish.sh` runs the whole pipeline: transcode videos, generate the
manifest + thumbnails, sync everything to R2, and build the static viewer.

```sh
export ASSET_BASE_URL="https://assets.samothraki.example"   # your R2 domain
export R2_BUCKET="r2:samothraki-holiday"                    # rclone remote:bucket
export CF_PAGES_PROJECT="samothraki-holiday"                # optional: auto-deploy
scripts/publish.sh
```

The generator is idempotent — thumbnails are only rebuilt when missing or
older than their source, so re-runs are fast.

`ASSET_BASE_URL` is baked into the WASM at compile time (via `option_env!`).
It defaults to root-relative (empty), so a bare `dx build` produces a bundle
that expects `photos/`, `thumbs/`, `media/`, and `manifest.json` served from
the same origin.

### Local viewer preview (no R2)

To preview the static viewer without R2, `scripts/setup.sh` builds it
root-relative and stages the assets next to the bundle:

```sh
scripts/setup.sh
python3 -m http.server -d target/dx/my-holiday/release/web/public 8080
```
